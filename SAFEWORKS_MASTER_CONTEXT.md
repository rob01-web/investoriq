# SafeWorks Master Context

Last updated: 2026-05-11

## Purpose
This is the single working memory file for SafeWorks.

Use this file at the start of each SafeWorks session to keep product direction, positioning, pricing, and current priorities aligned.

SafeWorks is not meant to replicate the Habitat M365 system 100 percent. Habitat proved the workflow. SafeWorks should keep the validated safety logic, then simplify the delivery and improve the product experience for SMBs.

## Product Promise
SafeWorks is a practical safety operating system for small and mid-sized businesses that replaces paper clipboards, scattered spreadsheets, and email follow-up with one simple app for:
- hazard and near-miss reporting
- safety inspections
- corrective actions
- investigations when needed
- open and closed lifecycle tracking
- reporting and export views
- photo and attachment capture
- location-based organization

The product should feel simple, fast, trustworthy, and operational. It should not feel like institutional EHS software.

## Target Customer
Primary audience:
- SMBs that need a better safety process
- teams still using paper, clipboard, or fragmented spreadsheets
- organizations that cannot afford enterprise safety software
- businesses that want a practical system without Microsoft tenant complexity

The commercial goal is a productized safety system with one clear offer, not a custom consulting project every time.

## V1 Feature Set
Keep V1 focused on the parts SMBs actually need:
- hazard / incident / near-miss reporting
- safety inspections
- corrective actions
- investigations
- open / closed status tracking
- dashboard for active work
- reporting and export views
- photo uploads
- reminders and follow-up
- basic roles and access control
- location-based organization
- mobile-friendly forms
- QR access to key forms

Avoid V1 bloat:
- no enterprise EHS sprawl
- no overly complex admin setup
- no per-client custom dashboard redesign by default
- no Microsoft 365 tenant dependency
- no workflow reinvention just to look different

## What To Keep From Habitat
The Habitat system proved the backbone. Keep these principles:
- inspections stay separate from hazards
- dashboards are for active work
- reporting views preserve open and closed history
- lifecycle close and reopen behavior matters
- corrective action text must stay separate from boolean action flags
- location naming and grouping must remain scalable
- operational dashboards and reporting views should stay separate

## What To Improve For SMBs
SafeWorks should be easier to sell and easier to use than the Habitat M365 build:
- simpler onboarding
- fewer fields up front
- cleaner mobile experience
- faster dashboard views
- better visual polish
- automated reminders
- clearer status flow
- no SharePoint or tenant setup burden for the customer

## Recommended Stack
Use a simple, low-cost stack:
- Supabase for data, auth, and storage
- Vercel for hosting
- lightweight edge/server logic only when needed

This keeps SafeWorks productized and avoids dependence on Microsoft tenants for each customer.

## Branding Direction
SafeWorks should feel related in quality to InvestorIQ, but not identical.

Recommended direction:
- operational and field-ready
- cleaner and more utilitarian than InvestorIQ
- slightly more industrial
- different font style from InvestorIQ
- restrained palette with one strong accent color

Suggested feel:
- deep slate, forest, steel, or safety-orange accents
- less editorial than InvestorIQ
- still polished and trustworthy

## Pricing Direction
The business model should be simple and standardized.

Suggested starting structure:
- one-time setup / implementation fee
- then a low monthly maintenance fee
- maintenance includes a small, clearly capped number of tweaks or changes each month

Practical starting point:
- use the Habitat sale as proof that a fixed implementation fee can work
- for SMBs, consider a flat upfront fee plus a low monthly fee, such as $49/month for light support
- define exactly what counts as a tweak so the monthly offer does not turn into open-ended custom consulting

Example product framing:
- Fixed setup fee: builds and launches the account
- Monthly fee: hosting, maintenance, and a limited number of small changes
- Anything beyond that becomes separate paid work
